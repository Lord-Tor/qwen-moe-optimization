# Доменная специализация MoE через градиентное смещение роутера

Метод точечного улучшения Mixture-of-Experts моделей на узких доменах **без дообучения весов** — только через смещение логитов роутера во время инференса. На `Qwen1.5-MoE-A2.7B` метод даёт **статистически значимый прирост +9–11 п.п. accuracy** на математических доменах MMLU. Проверка на второй архитектуре (`OLMoE-1B-7B`) показала, что эффект **архитектурно-зависим**: значимого переноса нет — что очерчивает границы применимости метода.

## Идея

В MoE модели каждый токен маршрутизируется к top-k экспертам через роутер (gate). Гипотеза: для конкретного домена существуют «полезные» эксперты, и если мягко подтолкнуть роутер к ним, точность на домене вырастет без изменения весов модели.

Пайплайн в три шага:
1. **Калибровка.** Прямой+обратный проход по ~40–100 примерам домена, сбор градиента `∂L/∂(router_logits)` по каждому слою. Эксперты с наибольшим отрицательным градиентом — те, усиление которых снижает loss.
2. **Смещение.** На инференсе forward-хук добавляет bias к логитам этих экспертов и **пересчитывает top-k** (критично: смещения одних логитов недостаточно, маршрут определяется индексами после top-k).
3. **Оценка.** One-pass next-token scoring на MMLU, сравнение с baseline через парный тест.

## Ключевые результаты

### Qwen1.5-MoE-A2.7B — метод работает и значим

| Домен | baseline | + bias (опт.) | Δ | McNemar p | random-control Δ |
|---|---|---|---|---|---|
| abstract_algebra | 31.0% | 42.0% | **+11.0** | 0.013 ✓ | +2.0 (нз) |
| college_mathematics | 31.0% | 40.0% | **+9.0** | 0.049 ✓ | 0.0 (нз) |
| high_school_mathematics | 28.9% | 38.5% | **+9.6** | 0.014 ✓ | +3.0 (нз) |
| formal_logic | 35.7% | 42.9% | +7.1 | 0.211 | +6.4 (нз) |

Прирост **значим на 3 из 4 доменов** (McNemar, парный тест), подтверждён бутстрап-CI, и **специфичен**: контроль со случайными экспертами той же геометрии незначим. На formal_logic эффект неотличим от случайного — контроль это выявил.

### Оптимум вмешательства

Зависимость Δ(множитель) имеет форму перевёрнутой U на всех доменах: малый множитель (×1–5) мягко подталкивает роутер и улучшает accuracy, большой (×10+) разрушает естественную маршрутизацию и роняет точность. Это объясняет, почему наивный агрессивный bias не работал.

### Цена специализации

Math-калибровка, применённая к нематематическим доменам: при рабочих множителях (×1–2) деградация права/биологии/истории в пределах |Δ|≤1.4 п.п. — **специализация локальна и почти бесплатна в рабочей зоне**. Обвал начинается только при ×10+, где и целевой домен уже сломан. Фактологические домены чувствительнее «рассуждательных» (moral_scenarios почти не страдает).

### Перенос на OLMoE-1B-7B

Метод технически переносится на другую MoE-архитектуру (64 эксперта, top-8, без shared expert) без изменения логики — хук, калибровка и инференс работают без правок. Однако **статистически значимого прироста нет ни на одном домене** (McNemar p>0.05 везде): bias чинит примерно столько же ответов, сколько ломает (например abstract_algebra: 18 исправлений против 16 поломок). Наибольшая тенденция — formal_logic (+8.7 п.п., 26 vs 15), но и она не достигает значимости на n=126 (p=0.117).

Это ключевой результат об **архитектурной зависимости** метода. Вероятное объяснение контраста с Qwen: у OLMoE более тонкая гранулярность экспертов (64/top-8 против 60/top-4) — каждый эксперт несёт меньше «доменной» специализации, поэтому смещение к отдельным экспертам даёт меньший эффект; плюс отсутствие shared expert меняет динамику смешивания. Вывод: эффективность градиентного смещения роутера зависит от устройства MoE, а не является универсальным свойством.

## Структура репозитория

```
src/
  utils.py                 промптинг, токенизация ответов, извлечение top-k роутера
scripts/
  build_gradient_bias.py   калибровка: сбор градиентов роутера -> конфиг смещений
  make_bias_from_grads.py   конфиги (normal / exclude-dominant) из сырых градиентов
  merge_grads.py            объединение градиентов доменов (общий math-конфиг)
  qwen_mmlu_biased.py       инференс со смещением + self-check + random-control
  qwen_mmlu_onepass.py      baseline-оценка MMLU
  multiplier_scan.py        скан Δ(множитель)
  stat_analysis.py          McNemar + бутстрап-CI
  aggregate_all.py          сводка результатов в JSON/CSV
  plot_figures.py           графики Δ(множитель), bias vs random
  analyze_cost.py           анализ цены специализации
  run_*.sh                  Slurm-обёртки
configs/                    JSON-конфиги смещений (генерируются)
results/summary/            сводки и графики (под git), сырые .jsonl — в .gitignore
```

## Установка и запуск

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# Калибровка + конфиг для домена
python scripts/build_gradient_bias.py --subject high_school_mathematics \
    --output configs/math_grad.json --save_raw_grads configs/raw/math.pt

# Инференс со смещением (self-check проверяет, что bias влияет на выход)
python scripts/qwen_mmlu_biased.py --subject high_school_mathematics \
    --bias_file configs/math_normal.json --bias_multiplier 5.0 --random_control

# Статистика и сводка
python scripts/stat_analysis.py --baseline_glob "results/*_baseline.jsonl" --biased_dir results/final
python scripts/aggregate_all.py --model_name Qwen1.5-MoE-A2.7B
python scripts/plot_figures.py --summary results/summary/Qwen1.5-MoE-A2.7B_summary.json
```

Полный воспроизводящий прогон: `sbatch scripts/run_night.sh` (Qwen), `sbatch scripts/olmoe_full.sh` (OLMoE).

## Воспроизводимость и инфраструктура

- Модели: `Qwen/Qwen1.5-MoE-A2.7B`, `allenai/OLMoE-1B-7B-0924`, fp16.
- Оценка: MMLU test, one-pass next-token logprob по токенам ` A/B/C/D`.
- Кластер Slurm, 2× TITAN RTX 24GB. Эксперты в режиме `eager` (контроль пиковой памяти).
- Каждый biased-прогон содержит self-check (инъекция реально меняет выход) и random-control (специфичность эффекта).

## Ограничения

- Прирост значим на 3/4 math-доменов Qwen; на formal_logic неотличим от случайного.
- На OLMoE-1B-7B значимого переноса нет — метод архитектурно-зависим.
- Эффект чувствителен к множителю (оптимум, не монотонность).
- Размер тестовых сплитов (n=100–270) ограничивает мощность теста — на OLMoE тенденция formal_logic могла бы достичь значимости на большей выборке.
- Калибровка на 40–100 примерах вносит шум в выбор экспертов.
